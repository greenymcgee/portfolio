## Description

Add the sticky action bar and wire up the Lexical editor. This PR also introduces the new `RichTextEditor` (which replaces the renamed `LegacyRichTextEditor` in name only — it is a different component with different responsibilities) and lifts the `LexicalComposer` to `EditPostForm`.

**`LexicalComposer` strategy (critical):**
The `ToolbarPlugin` must be a descendant of a `LexicalComposer`. Because the toolbar renders in `ActionBar` (above the editor DOM subtree), the `LexicalComposer` must be owned by their common ancestor: `EditPostForm`. `RichTextEditor` is rendered with no internal composer; it receives the composer context from the ancestor. See `decisions.md` → D5.

```
EditPostForm ('use client')
  └── LexicalComposer         ← owned here
        ├── ActionBar (sticky)
        │     ├── ToolbarPlugin   ← inside LexicalComposer ✓
        │     └── AutoSaveStatus
        └── RichTextEditor (new — no internal LexicalComposer)
```

**New `RichTextEditor`:**
- No internal `LexicalComposer` (it assumes one exists in its ancestor tree).
- No embedded toolbar (toolbar lives in `ActionBar`).
- `ToolbarPlugin` is re-exported from `globals/components/richTextEditor/index.ts` for use in `ActionBar`.

**`ActionBar`:**
- Sticky top bar.
- Contains: `ToolbarPlugin`, `AutoSaveStatus`, and placeholder slots for buttons (PRs 10–11).
- Receives `cancelDebounce` from `EditPostForm` as a prop (passed through to buttons in later PRs).

**`AutoSaveStatus`:**

| State | Display |
|-------|---------|
| `idle` | "Edited [date-fns phrase]" — `formatDistanceToNow(post.updatedAt, { addSuffix: true })` |
| `saving` | Spinner + "Saving..." |
| `saved` | "Saved" — persists until next save cycle |
| `error` | "Updates not saved" — destructive red |

Autosave errors route through `AutoSaveStatus` ("Updates not saved") and also fire a Sonner toast ("Post could not be saved").

## Technical Notes

- `LegacyRichTextEditor` (renamed in PR 1) creates its own `LexicalComposer` internally — that behavior is unchanged for its existing consumers. The new `RichTextEditor` does not.
- The `ToolbarPlugin` re-export must not break existing `LegacyRichTextEditor` consumers.
- See `decisions.md` → D5 (LexicalComposer strategy) and D14 (AutoSaveStatus / Sonner split).

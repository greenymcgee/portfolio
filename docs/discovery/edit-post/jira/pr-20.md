## Description

Keep `SiteNavbar` and `ClientSiteNavbar` mounted in the root `app/layout.tsx` and
restrict where the navbar shows by guarding it internally. `SiteNavbar` returns
`null` unless the pathname is `/` (home) or `/posts`. This hides the navbar on
`/login`, `/posts/[id]`, and `/posts/[id]/edit` while preserving the Work↔Blog
sliding-pill animation, which only transitions when the navbar stays mounted
across navigation.

This reverses the original plan (remove from the root layout and render opt-in at
each call site). See `decisions.md` → D52 (supersedes D47).

## Acceptance Criteria

- [ ] `SiteNavbar` returns `null` when `pathname` is neither `ROUTES.home` nor `ROUTES.posts`
- [ ] `SiteNavbar` renders on `/` and `/posts`
- [ ] Navbar is hidden on `/login`, `/posts/[id]`, and `/posts/[id]/edit`
- [ ] `SiteNavbar` and `ClientSiteNavbar` remain mounted in `app/layout.tsx` (no removal)
- [ ] Work↔Blog pill animation still transitions when moving between `/` and `/posts`
- [ ] `SiteNavbar` test covers the null case plus the `/` and `/posts` render cases
- [ ] `pnpm validate` passes

## Technical Notes

- The guard is an **allowlist** (`ROUTES.home`, `ROUTES.posts`), not a denylist. A new
  route is hidden by default and requires no edit to the navbar — this avoids the
  buried-exclusion-list problem D47 was trying to solve, without losing the animation.
- `ClientSiteNavbar`/`usePathname` are retained intentionally. Rendering the navbar
  per-page would remount it on each navigation and break the `transition-all` slider.
- `AdminMenu` still renders inside `SiteNavbar`. Post pages that need the admin menu
  reach it through their own page-level rendering, independent of the navbar's visibility.

## Description

Implement the `updatePost` server action and the full backend stack it depends on. This is the autosave entry point for the edit page. Also adds `CACHE_TAGS.post` and wraps `getPost` in `'use cache'` with the new tag. Fixes `deletePost` to revalidate `CACHE_TAGS.post`.

**Action → DTO → Service → Repository → `Result` flow:**

- `UpdatePostDto` validates `{ id, title, description, content }` using a Zod schema.
- `PostService.update` calls `authenticateAPISession()` then `authorizeUser(token, 'posts', 'update')`, then delegates to the repository.
- `PostRepository.update` writes `{ title, description, content }` to the `Post` row. **Never touches `publishedAt`.**
- `updatePost` action calls `.match()` on the result: on success, revalidates `CACHE_TAGS.post` and `CACHE_TAGS.posts`; on error, returns a typed error branch.

**`getPost` caching:**

```ts
'use cache'
cacheTag(CACHE_TAGS.post)
```

`CACHE_TAGS.post` is added to `globals/constants/cacheTags.ts`. `deletePost` is updated to call `revalidateTag(CACHE_TAGS.post)` (it currently only revalidates `CACHE_TAGS.posts`).

## Technical Notes

- `updatePost` never writes `publishedAt`. This is intentional — an autosave that fires after a publish must not clear the published state. See `decisions.md` → D20.
- `UniqueConstraintError` is returned (not thrown) — the `updatePost` action surfaces it to the save-state indicator and renders an inline error below the title input.
- Follow the existing `Action → DTO → Service → Repository → Result` pattern from `features/posts/actions/createPost.ts`.
- `getPost` caching uses a dedicated `CACHE_TAGS.post` tag so writing post A evicts only post A's detail cache, not the entire posts list. See `decisions.md` → D11.

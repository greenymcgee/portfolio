## Description

Implement the full publish/unpublish stack: server action, service method, repository method, and the `PublishUnpublishButton` UI component.

**Backend:**

`togglePostPublishedStatus` reuses `UpdatePostDto`. Two optional fields must be added to `UpdatePostDto` (and `updatePostSchema`) before this PR:

- `publishedAt` — the post's current `publishedAt` value from the DB, sent as a hidden input. Used to determine toggle direction: truthy → unpublish; falsy → publish. Never written to the DB directly.
- `redirectUrl` — the post detail page URL (`ROUTES.post(id)`), sent as a hidden input only on the publish path.

`PostService.togglePublishedStatus(dto)`:
- Calls `authenticateAPISession()` then `hasPermission(user, 'posts', 'publish')`.
- Derives `newPublishedAt` from `dto.params.publishedAt`: truthy → `null`; falsy → `new Date()`.
- Calls `PostRepository.togglePublishedStatus(dto, newPublishedAt)`.

`PostRepository.togglePublishedStatus(dto, newPublishedAt)`:
- Single `prisma.post.update` writing `title`, `description`, `content`, and `publishedAt` atomically.

`togglePostPublishedStatus` action:
- On publish success (`dto.params.publishedAt` was falsy): `revalidateTag(CACHE_TAGS.post)`, `revalidateTag(CACHE_TAGS.posts)`, then `redirect(dto.params.redirectUrl)`.
- On unpublish success (`dto.params.publishedAt` was truthy): `revalidateTag` both tags, return `{ ...params, status: 'SUCCESS' }`.
- Error branches mirror `updatePost` exactly — same structure including `unique-constraint` returning `threwUniqueConstraintError: true`.

See `decisions.md` → D40.

**`PublishUnpublishButton` UI:**

- **Disabled** when any of title, description, or content is empty.
- On submit: accesses `formRef.current` to read current form values; adds hidden inputs for `id`, `title`, `description`, `content`, `publishedAt`, and (when publishing) `redirectUrl`.
- **While in flight:** button shows **"Publishing..."** or **"Unpublishing..."** with a spinner.
- **On publish success:** redirect to `ROUTES.post(id)` (handled by the action). Sonner toast — "Success!" — appears on the post page after redirect (D39).
- **On unpublish success:** Sonner toast — "Success!"; label toggles in-place. No redirect (D39).
- **On publish failure:** Sonner toast — "Post could not be published" (D39).
- **On unpublish failure:** Sonner toast — "Post could not be unpublished" (D39).

See `decisions.md` → D3 (separate actions), D9 (Publish redirects; Unpublish is in-place), D40 (toggle contract).

## Technical Notes

- `publishedAt` determines toggle direction — truthy → unpublish, falsy → publish. It is never written to the DB by the DTO; the repository derives the new value from `newPublishedAt` passed by the service.
- `redirectUrl` is only present in the form on the publish path. The action redirects using this value on publish success.
- `PublishUnpublishButton` receives `formRef` and `cancelDebounce` as props from `EditPostForm` and calls `cancelDebounce` before submitting.
- `posts.publish` permission already exists in `POLICIES`. See `decisions.md` → D3.
- See `decisions.md` → D40 for full backend contract and rationale.

## Description

Add a fully Prisma-managed migration that makes two schema changes to the `Post` table:

1. Remove `@default("{}")` from `Post.content` — `createPost` will always supply a valid Lexical state via `createHeadlessBlogEditor()`. The default is a latent footgun: `{}` is not a valid Lexical document and would break `RichTextEditor` if it ever reached the editor.
2. Add `@unique` to `Post.title` — enforces title uniqueness at the database level. The timestamped placeholder approach (D8) means `createPost` never writes an empty title, and `UpdatePostDto` requires a non-empty title, so blank-title autosaves are rejected before reaching the DB.

Both changes are expressible in `schema.prisma`. No hand-authored SQL is needed.

**Workflow:**
1. Remove `@default("{}")` from `Post.content` in `schema.prisma`.
2. Add `@unique` to `Post.title` in `schema.prisma`.
3. Run `prisma migrate dev` — Prisma generates and applies both changes.

**Prisma-generated `migration.sql`:**
```sql
ALTER TABLE "Post" ALTER COLUMN "content" DROP DEFAULT;
CREATE UNIQUE INDEX "Post_title_key" ON "Post"("title");
```

## Technical Notes

- The timestamped placeholder title (e.g. `"Untitled — 2026-05-02 10:30:45"`) means two drafts created within the same second would collide. Acceptable at this scale.
- See `decisions.md` → D23 (unique constraint rationale) and D2 (content default removal).

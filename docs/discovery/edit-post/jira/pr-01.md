## Description

Add a Prisma-managed migration that makes two schema changes to the `Post` table:

1. Remove `@default("{}")` from `Post.content` â€” `createPost` will always supply a valid Lexical state via `createHeadlessBlogEditor()`. The default is a latent footgun: `{}` is not a valid Lexical document and would break `RichTextEditor` if it ever reached the editor.
2. Add a partial unique index on `Post.title WHERE title != ''` â€” allows multiple simultaneous empty-title drafts (the timestamped placeholder approach) while still enforcing title uniqueness for non-empty titles.

The migration is Prisma-managed with one hand-authored SQL statement. Prisma's DSL has no `WHERE`-clause support for partial unique indexes, so the index cannot be expressed in `schema.prisma` with `@unique`.

**Workflow:**
1. Remove `@default("{}")` from `Post.content` in `schema.prisma`.
2. Run `prisma migrate dev --create-only` â€” Prisma generates a timestamped migration directory with the schema-diff SQL.
3. Hand-edit the generated `migration.sql` to append the partial unique index.
4. Run `prisma migrate dev` to apply and record in `_prisma_migrations`.

**Final `migration.sql`:**
```sql
ALTER TABLE "Post" ALTER COLUMN "content" DROP DEFAULT;
CREATE UNIQUE INDEX "Post_title_key" ON "Post" (title) WHERE title != '';
```

Do **not** add `@unique` to `Post.title` in `schema.prisma` â€” Prisma would regenerate the migration without the `WHERE` clause.

## Description

Add an `unpublished` filter to the posts read path so an admin can view only unpublished posts. This feeds the Unpublished toggle in PR 12.

**Changes:**

- `FindAndCountPostsDto` gains an `unpublished?: string` field (query params are always strings).
- `PostService.findAndCount` accepts `{ page?: number; unpublished?: boolean }`. When `unpublished` is `true`, it calls `authorizeUser(token, 'posts', 'update')` before passing the flag to the repository. Unauthenticated or unpermissioned callers receive an error result — the flag is silently ignored for anonymous users.
- `PostRepository.findAndCount` adds `WHERE publishedAt IS NULL` when the `unpublished` flag is set.

**Why the permission check lives in the service, not in `getPosts`:**
`getPosts` is wrapped in `'use cache'` — auth logic must not cross the cache boundary. The cached function runs with no session context. Permission is checked in `PostService.findAndCount` before the cache is consulted. See `decisions.md` → D7.

## Acceptance Criteria

- [ ] `FindAndCountPostsDto` accepts `unpublished: 'true'` and parses it to `true`
- [ ] `FindAndCountPostsDto` treats missing/other `unpublished` values as `false` (not an error)
- [ ] `PostService.findAndCount` with `unpublished: true` and a valid admin session returns only posts where `publishedAt IS NULL`
- [ ] `PostService.findAndCount` with `unpublished: true` and no/invalid session returns an auth error (not unpublished posts)
- [ ] `PostService.findAndCount` without the flag returns the existing published-only behavior (unchanged)
- [ ] DB tests pass; `pnpm validate` passes

## Technical Notes

- The `unpublished` param is a string because Next.js search params are always strings. The DTO converts `'true'` → `true`.
- Do not put the permission check inside the `'use cache'`-wrapped `getPosts` function — auth logic must not cross the cache boundary.
- See `decisions.md` → D7 for the full rationale.
